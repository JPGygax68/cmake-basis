Introduction

The project template used at SBIA to instantiate new software development projects is documented herein. The purpose of this project template is twofold: At first, it standardizes the organization, build and test process, and automatic packaging of software packages developed at the lab. Second, it provides the developer with a predefined build configuration and convenient macros for tasks such as adding build targets and automatic generation of API documentation. It therefore makes use of CMake, a cross-platform, open-source build system. The template also contains various so-called Find modules which ease the task of resolving dependencies to external software libraries. The project template thus enables the software developer to concentrate as much as possible on the actual important part, the algorithmic details of their software, instead of the organization and configuration of the build system.

Requirements

The minimal requirements for using this project template is the availability of a CMake installation. In order to support the extraction of the project's current Subversion revision of an project using SVN for revision control, the svn command must be installed, which, however, is certainly given in this case anyways. Further, in order for the automatic file update feature to be able to merge the project's modified files with the latest version of the template files, Python has to be installed. In particular, the python interpreter and the standard modules os and sys, but also getopt have to be available. Otherwise, the project template can still be used, whereby the automatic file udpate will just be disabled.

Automatic File Update

One, on first sight extraordinary, feature of the project template is the mechanism of automatic file updates implemented as part of the configuration step of CMake. The reasons for this feature are as follows.
  1. A copy of CMake configuration files and modules which are common among all projects should still be part of each project such that the entire package is self-contained. If some of the CMake modules would be accessible at a system-wide location only, the distribution of the software packages becomes more difficult. In case of Find modules, a separate revision controlled SBIA package of such modules can be considered, though. A public distribution of such package is, however, required then. For instance, these CMake Find modules could be distributed as part of the SBIA Toolkit.
  2. The standard build configuration is maintained via the project template and each derived project at SBIA should adopt the modified standard as long as the changes do not interfere with the project specific part of the configuration of the build system. Note that the projects themselves shall and will mostly only be concerned about the implementations of particular algorithms and how to modularize these algorithms. Each component of the software is represented by one build target. The project developer shall ideally only be concerned about telling the build system which targets are to be build from which sources and which external dependencies have to be resolved. The template provides a build system which simplifies as much of this process as possible. As this SBIA specific build system may, however, continuously improved, in particular as long as the template is not entirely fixed, an update of projects which are already using the template is desired.
